### Q9: 数据字典生成方式
**问题**: 数据字典索引文件如何生成？

**回答**:
- A) 通过 MCP 工具从 metadata/ 自动生成（新增简单的索引生成工具）

**虚拟字段定义**:
- Lookup 字段的 `_name` 后缀虚拟字段（如 `new_primary_contact_id_name`）
- 计算字段
- 汇总/Rollup 字段

### Q12: 虚拟字段的完整定义
**问题**: 项目中的"虚拟字段"具体包括哪些？

**回答**: 需要搜索了解 Dataverse 中的虚拟字段。
- **举例**: Lookup 字段会有一个 name 后缀的虚拟字段
- **要求**: 在生成 dictionary 的时候不要生成虚拟字段

**Dataverse 虚拟字段类型**:
| 类型 | 模式/特征 | 示例 |
|------|-----------|------|
| Lookup 的 `_name` 后缀 | `_[a-z]+_name$` | `new_primary_contact_id_name` |
| 计算字段 | `is_calculated: true` | - |
| 汇总/Rollup 字段 | `aggregate_type` 存在 | - |

---

### Q13: 数据字典生成触发时机
**问题**: 数据字典何时生成？包含什么内容？

**回答**:
- **时机**: A) 在提交 Git 前通过 hook 生成
- **范围**: 只生成提交内容中更改到的范围，避免一次性生成过多
- **内容**:
  - **B)** 数据字典需要包括选项集
  - 选项集需要区分 Global 和字段本身的
  - 选项集需要包含**中文、英文、值**

### Q16: 开发工作流
**问题**: 当开发人员写插件时，如何让 LLM 知道读取数据字典？

**回答**:
- **A)** 在项目根目录放一个 `.claude/` 配置，指定 `docs/data_dictionary/` 为上下文，这正是开发者需要控制管理的上下文

## 关键决策汇总

### 1. 目录组织
| 决策 | 说明 |
|------|------|
| 按功能迭代组织 | `sources/features/feature-xxx/` |
| 转换器保留架构 | `transformers/` 暂不实现具体代码 |
| YAML 模板库 | `sources/library/` 存放可复用组件 |

### 2. 元数据管理
| 决策 | 说明 |
|------|------|
| 选项集存放 | 全局选项集独立在 `metadata/optionsets/` |
| 虚拟字段过滤 | 生成数据字典时排除 `_name` 后缀等虚拟字段 |
| 版本控制 | 源文件和 YAML 都纳入 Git |

### 3. 数据字典
| 决策 | 说明 |
|------|------|
| 生成方式 | Git pre-commit hook 自动生成 |
| 生成范围 | 仅生成变更的部分 |
| 内容包含 | 表结构 + 选项集（中英文值）+ 汇总索引 |
| LLM 集成 | 通过 `.claude/` 配置为上下文 |

### 4. 团队协作
| 决策 | 说明 |
|------|------|
| 所有权管理 | 内部约定，无需配置文件 |
| 冲突处理 | Git 工作流 + 约定 |
| 转换方式 | LLM/SKILL 辅助，手动触发 |
